# DriverPass System Analysis and Design Project

## Project Overview

This project was completed as part of a System Analysis and Design course and focuses on designing a system for DriverPass, a company that provides training to help students pass their DMV driving tests.

The client, Liam (the owner of DriverPass), required a system that could provide online learning resources, schedule in-person driving lessons, manage users and vehicles, and track student progress. The primary goal of the system is to reduce the high failure rate among students by improving access to training and creating a structured learning experience.

## Repository Contents

This repository includes two key documents that demonstrate the system analysis and design process:

* Business Requirements Document (BRD)
  Defines the system’s purpose, requirements, constraints, and user needs

* System Design Document (SDD)
  Includes UML diagrams (use case, activity, sequence, class) and technical requirements for the system

These documents show how business requirements are translated into a structured system design.

## What I Did Well

One of my strongest areas in this project was breaking down the system into clear components and representing them through UML diagrams. I was able to take a real-world problem and translate it into structured visual models that show how users interact with the system and how different components work together.

I also maintained alignment with the client’s needs throughout the design process, particularly in areas such as scheduling, user roles, and activity tracking.

## What I Would Improve

If I were to revise this project, I would focus on refining the UML diagrams by reducing unnecessary complexity and improving clarity in relationships between components.

I would also improve consistency across diagrams to ensure alignment in structure and level of detail.

## Interpreting User Needs

The system design was driven by the client’s primary concern: a high student failure rate. This need was translated into features such as practice testing, lesson scheduling, and progress tracking.

Considering user needs is critical because a system must solve real problems. If the design does not align with user goals, the system will not be effective.

## Approach to System Design

My approach to designing software begins with understanding the problem and identifying key users and system functions. From there, I break the system into components and model them using diagrams.

Key aspects of my approach include:

* Structuring the system clearly
* Mapping workflows logically
* Ensuring all components are connected and consistent

In the future, I will continue to use UML diagrams, requirement analysis, and iterative design to improve clarity and system effectiveness.
